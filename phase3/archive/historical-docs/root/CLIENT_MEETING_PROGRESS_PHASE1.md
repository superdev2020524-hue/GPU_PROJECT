# Phase 3 client meeting progress summary

This note is written against the pre-Phase 3 meeting direction in `addtion_require_2026_2_18.txt`. The main instruction from that meeting was clear: do not spend the early phase on cloud plumbing, advanced scheduling, or hardening first; prove the core functionality first. In practical terms, that meant getting a VM running Ollama to issue CUDA work through the mediation layer, execute that work on the physical H100, and return a real inference result back into the VM while keeping the virtualization layer transparent to the instance user.

That is the frame I have been working to throughout Phase 3. I kept the effort centered on the VM-to-mediator-to-physical-GPU path, because that was the shortest route to proving the architecture. The cloud-facing layer, tenant policy, and mature scheduling model were intentionally treated as follow-on work, exactly as discussed in the meeting. The work in this phase has therefore been less about adding surface features and more about forcing the full GPU execution path to behave correctly under real Ollama loads, then removing each blocker in the order it actually appeared.

The first major checkpoint was proving that the VM side behaved like a real GPU-enabled system rather than a mock path. That part is in place. Ollama has been brought up in CUDA mode inside the guest, the guest shim layer intercepts the CUDA calls, the VGPU-STUB transports those calls out of the VM, and the host mediator replays them against the physical H100. This is the core architectural promise from the meeting: the guest sees what looks like a GPU-backed environment, while the routing and arbitration stay inside the mediation layer.

The second checkpoint was getting past basic discovery and into real model-load behavior. That required several rounds of low-level debugging because the failures were not all in the same place. I had to correct service configuration drift that pushed Ollama back into CPU mode, fix library-resolution issues so the runner consistently used the intended CUDA and cuBLAS paths, and keep the guest-side shim and host-side mediator aligned as the execution path got deeper. This work was necessary before any inference result could be trusted, because a "GPU detected" log line by itself was not enough.

The third checkpoint was transport correctness. Earlier in the phase, some runs looked like GPU faults but were actually caused by transport-path problems underneath them. The most important of these was the shared-memory payload issue that produced zeroed host-to-device data in a path that should have been non-zero. That problem was traced to the stub-side bulk-source handling, not to Ollama itself. I fixed it by updating the vGPU stub and rebuilding the host QEMU package so the data path stopped selecting the wrong source and stopped forwarding all-zero payloads. That was a meaningful turning point, because it removed one of the false blockers that made later CUDA faults look worse than they were.

Another key step was separating runner crashes from earlier transport and allocation symptoms. I added crash-capture support, collected real core files from the VM, and correlated those with fresh mediator logs from the same runs. That made it possible to stop treating every `HTTP 500` or `exit status 2` as the same class of problem. In some cases the runner failure was just the visible outcome of an earlier CUDA failure. In other cases it exposed a genuine late-stage GGML/CUDA problem inside the model-load path. That distinction matters, because it changed the work from guesswork into targeted debugging.

There has also already been a direct end-to-end proof that the meeting’s primary functionality target is achievable. On a bounded Phase 3 run, `tinyllama` completed in GPU mode from inside the VM and returned a real `HTTP 200` response over the mediated path. That run demonstrated the essential Phase 1 behavior: GPU discovery in the guest, model load over the virtualization path, execution on the host H100, and a response returning cleanly to the VM. I do not treat that as "done" in the project-management sense yet, but it is important because it proves the architecture is real, not theoretical.

The reason I am not calling Phase 1 fully closed yet is that the current active blocker is now a late-stage stability issue rather than a foundational design problem. The active error at the moment is `E9`: a host-side `cuLaunchKernel` synchronization failure with `CUDA_ERROR_ILLEGAL_ADDRESS` on the GGML CUDA `soft_max_f32` path during deeper model execution on the repaired Test-10 baseline. The active error remains active because fresh repaired-baseline runs still reach deep GPU offload successfully and then terminate on that same late launch path. Current candidates, which are being tracked but not promoted ahead of the active blocker, are `E1` (historical invalid-image module-load failure), `E4` (other illegal-address class failures after GEMM work), `E5` (MMQ/graph-reserve abort path), `E8` (`0x00bc` capability-query anomaly), and `E10` (shared-memory registration fallback leading to BAR1 slowness on some cold-load runs). The reason `E9` stays active is that it is currently the earliest repeatable correctness-breaking failure on the repaired GPU baseline.

The evidence for that current position is stronger than it was earlier in the phase. On the repaired direct-service GPU baseline, the VM reaches CUDA initialization, offloads model layers, and gets well past early setup. The host mediator logs repeated successful launches before the first terminating fault appears. The first consistent terminating host fault is the `soft_max_f32` launch path, where `cuLaunchKernel` sync returns `rc=700`, and the guest then reports the matching CUDA error before the runner aborts. In other words, the stack is no longer failing at discovery, at first allocation, or at the original bulk-transfer path. It is failing much later, which is a sign of progress even though it is still a blocker.

One of the more important engineering observations from Phase 3 is that the remaining work is concentrated. The architecture is built. The mediation path exists. The guest-to-host GPU call flow is live. GPU-mode inference has already been demonstrated. The unresolved part is in the final stability and correctness layer: making the repaired baseline repeat the success cleanly, without falling into the late GGML/CUDA launch fault or cold-load regressions. That is a very different situation from being "early in development." It means the project is in finishing work on the critical execution path, not still trying to prove the concept.

For the client meeting, the most honest way to describe the current state is this: the Phase 1 objective has already been proven once in substance, but it is not yet in a repeatable enough state to be presented as closed. If Phase 1 is defined strictly as "can a VM running Ollama use the virtualized path to execute on the physical GPU and return a result," then that proof already exists. If Phase 1 is defined as "is that path now stable and repeatable on the active baseline without special handling," then there is still focused engineering work left.

I would describe the remaining development required for Phase 1 as moderate, but highly concentrated. It is not another broad build-out of the system. It is a finishing pass on the most difficult part of the path: the late-stage GPU execution correctness problem and the last bits of baseline repeatability. In practical terms, this is the kind of work that usually takes disciplined iteration rather than major new feature development. The bulk of the engineering has already been spent on making the stack real; what remains is removing the final blocker and then revalidating the path under controlled repeated runs.

## Current progress summary

Phase 3 has already delivered the hard part of the concept: the invisible mediation layer is real, Ollama can operate in GPU mode inside the VM, CUDA work can be forwarded to the host H100, and the stack has already produced a successful bounded inference result through that path. The remaining issue is not whether the design works. The issue is making the current repaired baseline reliably stay on the working path instead of terminating at the late `soft_max_f32` launch failure or regressing into cold-load instability.

## Comments on Server 1 and Server 2

From the Phase 3 evidence, `Server 1` is the machine that has been exercised most directly and most seriously. This path is tied to the H100 80GB baseline, and the evidence is concrete rather than speculative: the guest reports CUDA discovery at `compute=9.0`, the mediator has already replayed real CUDA allocations and transfers onto the physical H100, and there is already a bounded `HTTP 200` inference result on this path. In practical terms, that means `Server 1` is no longer at the concept stage. It is the active engineering path, and the remaining issue is not whether the server can participate in the architecture, but whether the late-stage execution path can be made stable and repeatable under deeper load. The current evidence-backed blocker on this server is the `soft_max_f32` illegal-address fault during deeper GGML CUDA execution, which is a correctness problem in the late execution path, not a sign that the server is unsuitable.

My comment on `Server 1`, speaking as an engineer, is that it is the stronger proof server today because it has already carried the project through the difficult parts: discovery, transport, host replay, and at least one successful end-to-end inference. That matters more than raw specification alone. An 80GB H100 is already large enough to prove the virtualization path and to support serious model-loading work. The evidence also shows that several of the hardest failures on this path were not caused by VRAM exhaustion at all. They were caused by transport correctness, fatbin compatibility, library alignment, and late kernel-execution faults. Scientifically, that distinction is important: adding more memory does not fix an invalid kernel image, a corrupted payload, or an illegal memory access in a launched kernel.

For `Server 2`, the current repository does not contain the same depth of direct Phase 3 run evidence that it does for `Server 1`. So the honest engineering position is this: I can comment confidently on the hardware implication of the approximately 96GB memory size, but I should not pretend I have the same log-backed execution history on that server unless we add its actual runs and traces. Purely from a systems point of view, moving from 80GB to about 96GB gives roughly 20% more memory headroom. That is meaningful for larger model residency, KV cache growth, allocator margin, and reducing pressure during cold-load or multi-tenant experiments. It should make Server 2 the more comfortable platform for capacity-sensitive testing.

That said, the extra memory on `Server 2` should be described correctly. It improves headroom, but it does not automatically solve the classes of problems that have dominated Phase 3. If the failure is a `cuModuleLoadFatBinary` rejection, a cuBLAS/cuBLASLt kernel-package mismatch, a shared-memory transport defect, or a late `cuLaunchKernel` illegal-address fault, then the additional 16GB does not remove the root cause. In other words, `Server 2` is likely better as a scale and margin platform, but it is not a substitute for finishing the correctness work already isolated on `Server 1`.

If I had to summarize both servers in one practical sentence for a client, I would say this: `Server 1` is the proof server because it has already demonstrated the path with real execution evidence, while `Server 2` is likely the better capacity server because the additional memory gives more operational headroom, but it still needs the same software correctness work to benefit fully from that hardware advantage.

## Recommended meeting wording

If this needs to be summarized in one sentence during the client call, I would put it this way: Phase 3 has already proven the VM-to-physical-GPU execution path the client asked for, and the remaining work to finish Phase 1 is now concentrated in final stability and repeatability on the active baseline, not in core architecture or feature build-out.
